# v0.23.0 — Bilingual (Chinese + English) brain support

## What ships

- `src/core/tokenizer.ts` — single source of truth for CJK detection,
  word segmentation (via `Intl.Segmenter`), token counting, and pinyin
  transliteration (via `pinyin-pro`).
- Schema migration v30 (`cjk_pre_segmented_columns`):
  - Adds `pages.title_segmented`, `pages.compiled_truth_segmented`,
    `pages.timeline_segmented` (TEXT, weights A/B/C).
  - Adds `content_chunks.chunk_text_segmented`,
    `content_chunks.doc_comment_segmented` (TEXT, weights B + A).
  - Replaces `update_page_search_vector()` and
    `update_chunk_search_vector()` with bilingual shape — appends a
    `to_tsvector('simple', <segmented>)` channel per existing English
    channel.
- `searchKeyword` + `searchKeywordChunks` (both engines) build a bilingual
  tsquery: `websearch_to_tsquery('english', q) || plainto_tsquery('simple',
  segmentText(q))`. Latin-only queries are byte-for-byte unchanged (empty
  segmented query → empty plainto_tsquery → no-op OR).
- `putPage` and `upsertChunks` (both engines) populate the segmented columns
  via `segmentText()` before insert/update.
- Slug pipeline: `slugifySegment`, `slugifyEntity`, and the `makeResolver`
  `norm()` helper pinyin-transliterate Han ideographs first so a CJK file
  path like `companies/百度.md` becomes the slug `companies/baidu` instead
  of an empty segment. ASCII-only slug behavior is preserved for portability.
- Chunkers (`recursive.ts`, `semantic.ts`) accept CJK sentence/clause
  punctuation (`。！？，、；：`) as delimiters and use `countTokens` (Latin
  word count + segmented CJK word count) for chunk sizing.
- Intent classifier (`src/core/search/intent.ts`) recognizes Chinese
  temporal/event/entity keyword patterns alongside the existing English
  ones.

## What the agent should do post-upgrade

The orchestrator runs the schema migration automatically — that's the
required mechanical step. Two cases below.

### 1. Run the schema migration

```bash
gbrain apply-migrations --yes
```

Adds the 5 new segmented columns + replaces the two FTS triggers. Idempotent.
On a fresh PGLite install this is a no-op; on an upgraded Postgres/PGLite
brain it lands the columns and replaces the triggers in a single
transaction. Existing rows have NULL segmented columns post-migration —
the english channel keeps working, so existing search behavior is
preserved byte-for-byte.

### 2. (Optional) Backfill existing CJK pages

If your brain has existing pages with Chinese content and you want them
findable via Chinese keyword search, re-write them through `putPage` so
the segmented columns get populated:

```bash
gbrain sync --full
```

`sync --full` walks every markdown file and re-imports through
`putPage()`, which populates `title_segmented`, `compiled_truth_segmented`,
and `timeline_segmented` automatically. The trigger refires on UPDATE and
rebuilds `search_vector` with both english and simple channels.

Brains that are English-only can skip this step entirely — the english
channel is unchanged, no CJK content to index.

### 3. Verify

```bash
# Pages have segmented columns populated for CJK content:
gbrain query "百度"          # searches via the simple channel
gbrain query "OpenAI"        # English regression check, unchanged

# Schema:
gbrain doctor --fast
```

## Known limitations (v1 scope)

- **Slug backfill for pre-existing CJK pages.** Slugs are stable IDs;
  rewriting them breaks wiki links + FKs. Pinyin transliteration applies
  to NEW pages only. Existing pages whose slugs were stripped to empty
  segments (rare, since they would have failed to ingest cleanly) keep
  their current slugs.
- **Hiragana / Katakana / Hangul slugs** fall through to the existing
  ASCII-strip. Han ideographs are the only script transliterated in v1.
- **`timeline_entries.summary/detail` text** isn't segmented (gathered
  inline by the page trigger via `string_agg`). Page-level + chunk-level
  coverage handles 99% of search needs.
- **`Intl.Segmenter` word boundaries** are heuristic. ICU recognizes
  common terms (`百度` → one word, `张三` → one word) but splits some
  others (`腾讯` → `腾` + `讯`). Users can manually rename slugs when
  segmentation produces something unwieldy.
- **No custom dictionaries.** `Intl.Segmenter` ships with ICU's CJK
  breaker; domain-specific terms (jargon, brand names ICU doesn't
  recognize) get character-level segmentation. Layering `@node-rs/jieba`
  for higher-quality segmentation is a future change with build-pipeline
  implications.
